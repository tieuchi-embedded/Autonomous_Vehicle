\paragraph{I2C (Inter-Integrated Circuit)}

I2C is a two-wire synchronous serial protocol using a shared clock line (SCL) and a bidirectional data line (SDA).
Communication is initiated by a START condition (SDA falls while SCL is high), followed by a 7-bit device address,
a read/write bit, an ACK from the slave, data bytes, and a STOP condition (SDA rises while SCL is high).
In this project, I2C1 (PB8=SCL, PB9=SDA) communicates with BNO055 (\texttt{0x28}) and AS5600 (\texttt{0x36}) at 100\,kHz.

\paragraph{PWM (Pulse Width Modulation)}

PWM generates a periodic digital signal with fixed frequency and variable duty cycle.
The STM32 timer compares the counter value against a capture/compare register (CCR);
output is high when \texttt{CNT < CCR} and low otherwise.
PWM is used to control the servo steering angle and to send throttle commands to the ESC.
Typical servo period is 20\,ms (50\,Hz); pulse width 1--2\,ms maps to 0°--180°.

\paragraph{UART (Universal Asynchronous Receiver/Transmitter)}

UART is an asynchronous serial protocol with no shared clock.
A frame consists of: 1 start bit (low), 8 data bits (LSB first), optional parity, and 1--2 stop bits (high).
Both ends must agree on the same baud rate. This project uses 115200\,baud, 8N1 (no parity, 1 stop bit).
The STM32 transmits sensor data to the Raspberry Pi 5 over UART in ASCII frames:
\texttt{@IMU:yaw,pitch,roll;;\textbackslash r\textbackslash n} and \texttt{@RPM:rpm;;\textbackslash r\textbackslash n}.

\paragraph{RCC (Reset and Clock Control)}

The RCC peripheral on the STM32F401 manages system clock configuration and peripheral clock gating.
The internal HSI oscillator (16\,MHz) or external HSE crystal can be fed into the PLL to generate
the system clock (SYSCLK). On the STM32F401 NUCLEO, SYSCLK is configured to 84\,MHz.
Peripheral buses APB1 (max 42\,MHz) and APB2 (max 84\,MHz) are derived via prescalers.
Each peripheral must have its bus clock explicitly enabled via \texttt{RCC\_APBxENR} before use.

% ---- Timer ----
\paragraph{Timer}

STM32 general-purpose timers (TIM2--TIM5) use an up-counter that increments each tick.
The tick frequency is \(\frac{f_{APB} \times 2}{PSC+1}\); the counter resets at \texttt{ARR} (Auto-Reload Register),
producing an overflow period of \(\frac{ARR+1}{f_{tick}}\).
In this project:
\begin{itemize}
  \item \textbf{TIM1/TIM4} — PWM output for servo and ESC (\(f=50\,\text{Hz}\), \(ARR=19999\), \(PSC=83\))
  \item \textbf{TIM2} — encoder interface mode, counts quadrature edges from AS5600 (via GPIO simulation)
  \item \textbf{TIM3} — periodic interrupt for UART transmit scheduling (1\,kHz)
\end{itemize}

\subsubsection{Actuator}

The vehicle uses two actuators driven by PWM signals from the STM32 at 50\,Hz:
a position servo for steering and a brushless motor with ESC for propulsion.
Both are commanded via \texttt{TIM2} (PA0/PA1, AF1), configured with \(PSC=15\), \(ARR=19999\)
so that CCR units map directly to microseconds (1\,tick = 1\,\(\mu\)s).

% ---- Steering Servo ----

The front axle is steered by a standard RC position servo.
The STM32 writes the desired pulse width to \texttt{TIM2->CCR2} (PA1, TIM2\_CH2).
Center position corresponds to 1500\,\(\mu\)s; each degree of steering offset maps to
10\,\(\mu\)s, giving an effective range of \(\pm40°\) (1100--1900\,\(\mu\)s).

The rear axle is driven by a brushless DC (BLDC) motor connected through a
7:1 gear reduction to the rear wheels (wheel radius 6.5\,cm).
Speed is commanded via an Electronic Speed Controller (ESC) that accepts the same
50\,Hz PWM protocol as a servo: 1500\,\(\mu\)s = neutral/brake, values below 1500 reverse,
values above 1500 drive forward.

\begin{table}[H]
\centering
\begin{tabular}{ll}
\hline
\textbf{Property} & \textbf{Value} \\
\hline
Motor type      & Brushless DC (BLDC) \\
Speed controller & ESC, PWM 50\,Hz \\
Neutral / brake  & 1500\,\(\mu\)s \\
Forward range    & 1500--2000\,\(\mu\)s \\
Reverse range    & 1500--1000\,\(\mu\)s \\

Gear ratio      & 7:1 (motor to wheel) \\
Wheel radius    & 6.5\,cm \\

\end{table}

The Brain publishes \texttt{ControlCmd.rpm} \(\in [-500, +500]\) (positive = forward).
The STM32 calls \texttt{Servo\_Speed\_Set\_RPM(rpm)}, writing
\(1500 - \lfloor rpm \rfloor\) to \texttt{TIM2->CCR1}.
Motor RPM feedback is read back via the AS5600 encoder on the rear axle and
reported to the Brain in the \texttt{@RPM:value;;\textbackslash r\textbackslash n} UART frame,
where actual wheel speed is recovered as:
\[
  v = \frac{rpm_{\text{motor}}}{7} \times 2\pi \times 0.065 \;\text{[m/s]}
\]

\paragraph{Inter Process Communication}

Inter-Process Communication (IPC) refers to mechanisms that allow independent OS processes to exchange data.
On Linux, the kernel provides several IPC primitives, each with different trade-offs in latency, throughput,
ordering guarantees, and persistence.

\begin{table}[H]
\centering
\begin{tabular}{lp{3.8cm}p{3.8cm}l}
\hline
\textbf{Mechanism} & \textbf{Characteristics} & \textbf{Use case} & \textbf{Used in project} \\
\hline
POSIX Message Queue
  & Kernel-buffered, ordered FIFO queue; fixed max depth (\texttt{msg\_max}); supports blocking/non-blocking send/recv
  & Low-rate, queued messages where every message must be delivered
  & Yes — MQ transport \\
\addlinespace
Shared Memory (SHM)
  & Fastest IPC: processes map the same physical page; no kernel copy; requires explicit synchronization
  & High-rate data where only the latest value matters (sensor frames, state)
  & Yes — SHM latest-wins \\
\addlinespace
Unix Domain Socket
  & Stream or datagram socket over a file path; reliable, ordered; higher latency than SHM
  & Bidirectional communication between local processes
  & Not used \\
\addlinespace
TCP Socket
  & Network socket; works across machines; adds TCP overhead
  & Remote monitoring, cross-host communication
  & Not used \\
\hline
\end{tabular}
\caption{Linux IPC mechanism comparison}
\end{table}

This project implements two transports inside \texttt{libipc}:

\begin{itemize}
  \item \textbf{POSIX MQ} (\texttt{mq\_transport.c}): wraps \texttt{mq\_open}/\texttt{mq\_send}/\texttt{mq\_receive}.
        Queue depth is clamped to the system limit (\texttt{/proc/sys/fs/mqueue/msg\_max}, default 10).
        Used for topics where every message must be consumed (e.g.\ \texttt{CONTROL\_CMD}).
  \item \textbf{Shared Memory} (\texttt{shm.c}): publisher creates a named SHM object via \texttt{shm\_open} +
        \texttt{ftruncate}, writes a fixed-size slot protected by a \texttt{pthread\_rwlock}.
        Subscriber opens (does \emph{not} create) the same object and reads the latest value.
        Used for high-rate sensor data where stale frames are discarded (e.g.\ \texttt{CAMERA\_FRAME}, \texttt{EGO\_STATE}).
\end{itemize}

\paragraph{Pub/Sub Model}

The project adopts a publish/subscribe pattern on top of the two transports.
Each logical data stream is identified by a \emph{topic}; nodes are decoupled — a publisher does not know
who is listening, and a subscriber does not know the publisher's identity.

\begin{itemize}
  \item \textbf{Node} --- an independent OS process that owns one or more publishers or subscribers.
        Each node links against \texttt{liblibipc} and calls the C API directly.
  \item \textbf{Topic} --- a named channel identified by a \texttt{TopicId} enum value (e.g.\
        \texttt{CAMERA\_FRAME=1}, \texttt{SERIAL=2}, \texttt{EGO\_STATE=3}, \texttt{LANE\_STATE=4}, \texttt{CONTROL\_CMD=5}).
        Each topic has a fixed payload type, transport kind (MQ or SHM), and drop policy, declared in \texttt{topic.c}.
  \item \textbf{Publish} --- a node calls \texttt{ipc\_publish()} with a topic ID and a pointer to a POD struct.
        The bus layer routes the write to the correct transport. The publisher must start before any subscriber.
  \item \textbf{Subscribe} --- a node calls \texttt{ipc\_subscribe()} then \texttt{ipc\_recv()} (blocking or timed).
        For SHM topics the subscriber receives the latest written value; for MQ topics it dequeues in FIFO order.
\end{itemize}

\paragraph{Drop Policy}

Drop policy defines what happens when a new message arrives and the buffer is full, or when a subscriber
reads slower than the publisher writes.

\begin{table}[H]
\centering
\begin{tabular}{lp{5cm}l}
\hline
\textbf{Policy} & \textbf{Behavior} & \textbf{Transport} \\
\hline
\texttt{NEW}   & Drop the \emph{incoming} message if buffer full; keep existing messages & MQ \\
\texttt{OLD}   & Drop the \emph{oldest} message in buffer to make room for the new one  & MQ \\
\texttt{NEVER} & Latest write always overwrites the single slot; subscriber always gets newest value & SHM \\
\hline
\end{tabular}
\caption{Drop policies in \texttt{libipc}}
\end{table}

In this project all SHM topics use \texttt{NEVER} (latest-wins); MQ topics use \texttt{NEW} (block new if full).

\paragraph{C API / C++ Binding}

The C API (\texttt{libipc/include/ipc/bus.h}) exposes four functions:
\begin{itemize}
  \item \texttt{ipc\_publish\_open(TopicId)} --- allocate publisher handle, create SHM/MQ object.
  \item \texttt{ipc\_publish(handle, const void *data)} --- write one message.
  \item \texttt{ipc\_publish\_close(handle)} --- release handle, unlink SHM/MQ.
  \item \texttt{ipc\_subscribe\_open(TopicId)} / \texttt{ipc\_recv(handle, void *out, int timeout\_ms)} / \texttt{ipc\_subscribe\_close}.
\end{itemize}

The C++ binding (\texttt{libipc/bindings/cpp/include/ipc/bus.hpp}) wraps these into RAII templates:
\texttt{ipc::Publisher<T>} and \texttt{ipc::Subscriber<T>}, where \texttt{T} is the message struct type.
Constructors call \texttt{open}; destructors call \texttt{close} automatically.

\paragraph{Python Binding}

The Python binding (\texttt{libipc/bindings/python/ipc.py}) uses \texttt{ctypes} to call the shared library
\texttt{liblibipc.so} at runtime. Message structs are defined in \texttt{ipc\_schema.py} as
\texttt{ctypes.Structure} subclasses mirroring the C headers in \texttt{messages/}.
Usage pattern:
\begin{enumerate}
  \item \texttt{import ipc, ipc\_schema} (after inserting the bindings path into \texttt{sys.path}).
  \item \texttt{pub = ipc.Publisher(ipc\_schema.TopicId.LANE\_STATE)} --- open publisher.
  \item Fill a \texttt{ipc\_schema.LaneState()} struct and call \texttt{pub.publish(msg)}.
  \item \texttt{sub = ipc.Subscriber(ipc\_schema.TopicId.EGO\_STATE)}; \texttt{msg = sub.recv(timeout\_ms=100)}.
\end{enumerate}

\paragraph{Camera}

The camera module is the entry point of the visual pipeline. It acquires image
frames and publishes them to a shared-memory topic so downstream vision nodes
(lane detection, object detection) can read the latest frame without copying.
During development the real camera is replaced by a video simulator
(\texttt{camera\_sim}) that decodes an MP4 file and emits frames at the
source frame rate; the hardware capture node (libcamera) is planned but not yet
implemented.

\begin{table}[H]
\centering
\begin{tabular}{lll}
\hline
\textbf{Node} & \textbf{Subscribe} & \textbf{Publish} \\
\hline
\texttt{camera\_sim} & --- (reads MP4 file) & \texttt{CAMERA\_FRAME} (SHM) \\
\hline
\end{tabular}
\caption{Camera node communication}
\end{table}

\textit{Algorithm theory.}
Video decoding extracts successive frames from an encoded stream; spatial
resizing maps each frame onto a fixed grid using bilinear interpolation. The
frame is then exposed through shared memory using a \emph{latest-wins} policy:
the publisher overwrites a fixed slot and the subscriber always reads the most
recent frame, which is the correct semantic for high-rate sensor data where
stale frames are useless.

\textit{Application in project.}
Each frame is resized to $320 \times 180$ BGR (3 channels, \texttt{stride} $=960$
bytes, \texttt{fmt}$=0$) and written to \texttt{CAMERA\_FRAME} (SHM, depth $=3$,
drop policy NEW). The payload size is \texttt{sizeof(CameraFrame)} plus
\texttt{stride}$\times$\texttt{height} pixel bytes. The simulator loops the
video on end-of-file and sleeps to match the source frame rate (default
30\,fps).

\paragraph{Lane Detection}

The lane-detection module receives a camera frame and estimates the vehicle's
position relative to the lane. It outputs two scalars: the \emph{heading error}
(angle between the lane tangent and the vehicle's longitudinal axis) and the
\emph{lateral offset} (signed distance from the lane center). These feed the
lateral controller for lane following.

\begin{table}[H]
\centering
\begin{tabular}{lll}
\hline
\textbf{Node} & \textbf{Subscribe} & \textbf{Publish} \\
\hline
\texttt{lane} & \texttt{CAMERA\_FRAME} (SHM) & \texttt{LANE\_STATE} (SHM) \\
\hline
\end{tabular}
\caption{Lane node communication}
\end{table}

\textit{Algorithm theory.}
The pipeline combines several classical computer-vision techniques:
\begin{itemize}
  \item \textbf{Inverse Perspective Mapping (IPM)} --- a homography that warps the
        forward-facing camera view into a top-down (bird's-eye) view, in which
        parallel lane lines remain parallel and curvature is measurable.
  \item \textbf{HLS color thresholding} --- converting to the HLS color space and
        thresholding the Lightness (L) channel isolates bright lane markings
        from the dark road surface.
  \item \textbf{Histogram peak detection} --- summing white pixels per column over
        the lower half of the image locates the left/right lane bases.
  \item \textbf{Sliding-window tracking} --- starting from each base, a vertical
        stack of windows follows the lane upward, re-centering on the pixel mean.
  \item \textbf{Least-squares line fit} --- the collected pixels are fit to a line
        $x = b\,y + c$ via the normal-equation / SVD solver.
  \item \textbf{Exponential moving average (EMA)} --- temporal smoothing reduces
        frame-to-frame jitter in the output.
\end{itemize}

\textit{Application in project.}
The node runs the following pipeline on every frame:

\hline
\textbf{Stage} & \textbf{Parameter} & \textbf{Value} \\
\hline
IPM source (top)    & \texttt{TL\_X}, \texttt{TR\_X}, \texttt{TOP\_Y} & $0.20$, $0.80$, $0.50$ \\

After fitting the lane line $x = b\,y + c$, the heading error is the angle of
the line with respect to the vertical axis, and the lateral offset is the
horizontal distance between the lane reference and the image center (the
vehicle body), scaled to meters:
\[
  \text{heading\_err} = \arctan(b),
  \qquad
  \text{lateral\_offset} = \bigl(x_{\text{lane}}\big|_{y=h-1} - \tfrac{w}{2}\bigr)\cdot \texttt{PIXEL\_TO\_M}.
\]
The right lane is preferred; when both lanes are detected the reference $x$ is
the midpoint of the two fitted lines, otherwise the node falls back to the left
lane. The raw result is then filtered:
\[
  o_t = \alpha\, o_{t-1} + (1-\alpha)\, o_t^{\text{raw}}, \qquad \alpha = 0.7,
\]
applied independently to heading error and lateral offset. On detection
failure the previous smoothed value is held.

\paragraph{State}

The state module fuses the raw serial telemetry from the STM32 into a compact
ego-vehicle state. It converts the IMU yaw angle and the motor RPM into the
vehicle's heading and linear speed, which localization and control consume.

\begin{table}[H]
\centering
\begin{tabular}{lll}
\hline
\textbf{Node} & \textbf{Subscribe} & \textbf{Publish} \\
\hline
\texttt{state} & \texttt{SERIAL} (MQ) & \texttt{EGO\_STATE} (MQ) \\
\hline
\end{tabular}
\caption{State node communication}
\end{table}

\textit{Algorithm theory.}
Linear speed is derived from the motor angular velocity through the drivetrain:
dividing motor RPM by the gear ratio yields wheel RPM, and multiplying by the
wheel circumference converts to linear velocity. The heading angle is taken
directly from the on-chip fused yaw of the IMU.

\textit{Application in project.}
With a gear ratio of $7{:}1$ and wheel radius $6.5$\,cm, the node computes:
\[
  \text{angle} = \text{yaw}, \qquad
  \text{speed} = \frac{\text{rpm}}{7} \cdot 2\pi \cdot \frac{6.5}{60}\;\,[\text{cm/s}].
\]
The published \texttt{EgoState} also carries \texttt{time\_ms}, the milliseconds
elapsed since the node started.
